# はじめに - なぜドキュメントが重要なのか

## 本書の目的

技術ドキュメントは、ソフトウェア開発において**コードと同じくらい重要**です。しかし、多くの開発者がドキュメント作成に苦手意識を持っています。

本書は、以下のような疑問に答えます：

- 何をどこまで書けば良いのか
- 実測データや検証結果をどう扱うべきか
- 読者に誤解を与えない表現とは
- チームで持続可能なドキュメント戦略とは

**最も重要な原則は「誠実性」と「正確性」です。**

---

## なぜドキュメントが必要なのか

### コードだけでは伝わらないこと

優れたコードは自己文書化されていると言われますが、コードだけでは伝えられない情報があります：

```typescript
// ✅ コードは「何をするか」を示す
export function calculateDiscount(
  price: number,
  customerType: 'regular' | 'premium' | 'vip'
): number {
  const rates = {
    regular: 0.05,
    premium: 0.10,
    vip: 0.15
  };

  return price * (1 - rates[customerType]);
}
```

**コードが示すこと：**
- 関数名、引数、戻り値の型
- 実装の詳細（割引率の計算ロジック）

**コードが示さないこと：**
- なぜこの割引率なのか（ビジネス要件）
- いつ導入されたのか（歴史的背景）
- 将来的に変更される可能性はあるか（設計判断）
- どのような状況で使うべきか（ユースケース）

これらの情報は、**ドキュメントでしか伝えられません。**

---

### ドキュメントの価値

#### 1. オンボーディング時間の短縮

新しいメンバーがプロジェクトに参加する際、ドキュメントがあれば：

- プロジェクトの全体像を素早く把握できる
- セットアップ手順が明確になる
- よくある質問への回答がすぐ見つかる

一般的に、優れたドキュメントがあるプロジェクトでは、新メンバーのオンボーディングにかかる時間が短縮される傾向があります。

#### 2. コミュニケーションコストの削減

ドキュメントがない場合：

```
新メンバー: 「このAPIの認証方式は何ですか？」
→ チャットで質問 → 先輩が回答 → 15分の時間
```

ドキュメントがある場合：

```
新メンバー: API仕様書を読む → 5分で理解
```

同じ質問が5人から来た場合、ドキュメントの有無で **75分 vs 25分** の差が生まれます。

#### 3. 知識の民主化

ドキュメントは、特定の人だけが持つ暗黙知を明示知に変換します：

- 「この設計判断は誰が決めたんだっけ？」
- 「なぜこのライブラリを選んだの？」
- 「この制約は何のため？」

これらの質問に対する答えが **ADR（Architecture Decision Records）** などのドキュメントに記録されていれば、誰でもアクセスできます。

#### 4. 将来の自分への贈り物

3ヶ月後、6ヶ月後の自分は、今の設計判断を覚えていません。

```typescript
// ❌ コメントなし
if (user.lastLoginAt &&
    Date.now() - user.lastLoginAt.getTime() > 90 * 24 * 60 * 60 * 1000) {
  await archiveUser(user.id);
}
```

```typescript
// ✅ コメントあり
// 90日間ログインがないユーザーはアーカイブする
// - GDPR対応のため（2023年4月導入）
// - アーカイブ後も復元可能
// - 詳細: docs/user-archiving.md
if (user.lastLoginAt &&
    Date.now() - user.lastLoginAt.getTime() > 90 * 24 * 60 * 60 * 1000) {
  await archiveUser(user.id);
}
```

ドキュメントは、**未来の自分へのメッセージ**です。

## 本書の最重要原則

### 誠実性と正確性

技術ドキュメントを書く上で、**最も重要なのは読者の信頼**です。

#### ❌ 絶対にやってはいけないこと

**1. 実測していないデータを「実測」として書く**

```markdown
❌ 悪い例:
実測したところ、メモリ使用量が1,240MB → 85MBに削減されました（93%削減）。
```

このように書くと、読者は「著者が実際に測定した結果だ」と受け取ります。実際には測定していないのに測定したように書くことは、**読者への欺瞞**です。

**2. 架空の具体的数値を事実のように書く**

```markdown
❌ 悪い例:
このプロジェクトで、レスポンスタイムが5.2秒から1.1秒に改善しました。
```

具体的な数値（5.2秒、1.1秒）を書くと、「実際にあったプロジェクトでの実測値」と受け取られます。

**3. 「〜したところ」「検証した結果」という表現**

```markdown
❌ 悪い例:
本番環境で試したところ、パフォーマンスが大幅に向上しました。
検証した結果、エラー率が78%削減されました。
```

これらの表現は、「著者が実際に行った検証」と読者に誤解させます。

#### ✅ 正しい書き方

**1. 理論・原理を説明する**

```markdown
✅ 良い例:
Stream処理は、データ全体をメモリにロードせず、チャンク単位で処理するため、
メモリ使用量を大幅に削減できます。

理論的には、以下のような効果が期待できます：
- ピークメモリの削減
- GC圧力の低減
- 大容量データの安定的な処理
```

**2. 公式ドキュメント・ベンチマークを引用する**

```markdown
✅ 良い例:
Node.js公式ドキュメント（https://nodejs.org/api/stream.html）によると、
Streamは大容量ファイル処理に適したAPIです。

Fastify公式ベンチマーク（https://fastify.dev/benchmarks/）によると：
- Fastify: 78,513 req/sec
- Express: 14,200 req/sec

※ 環境や条件により結果は異なります
```

**3. 一般的な傾向として記載する**

```markdown
✅ 良い例:
一般的に、LRUキャッシュを導入すると、以下のような効果が期待できます：
- 頻繁にアクセスされるデータへの高速アクセス
- メモリ使用量の制限
- 古いデータの自動削除

実際の効果は、アクセスパターンやデータ構造により異なります。
```

**4. 想定されるケースとして書く**

```markdown
✅ 良い例:
想定されるケース：
10万件のユーザーデータを処理する場合、以下のような実装が推奨されます：

（実装例）

この実装により、理論的には以下のような効果が期待できます：
- メモリ使用量の安定化
- 処理時間の短縮（データ量により異なる）
```

### なぜ誠実性が重要なのか

#### 読者の信頼を失うリスク

技術者にとって、**信頼性**は最も重要な資産です。

一度「この人の情報は信頼できない」と思われると：

- 記事が読まれなくなる
- 本が売れなくなる
- 技術コミュニティでの評判が下がる
- 就職・転職にも影響する可能性

**誠実で正確な情報を提供することが、長期的な成功につながります。**

#### 実例：信頼を損なう表現

```markdown
❌ 問題がある例:

# 革命的な新手法で10倍高速化！

この新しい実装パターンを試したところ、驚異的な結果が出ました：

Before:
- 処理時間: 8.3秒
- メモリ使用量: 1,240MB
- エラー率: 12%

After:
- 処理時間: 0.8秒（10倍高速化！）
- メモリ使用量: 85MB（93%削減！）
- エラー率: 2.6%（78%改善！）

この手法を本番環境で運用して3ヶ月、トラブルは一度もありません。
```

**何が問題か:**
- 具体的な数値が「実測値」に見える
- 「試したところ」が「実際に試した」と読める
- 「本番環境で運用」が実在するプロジェクトに見える
- 「10倍高速化」が保証されているように見える

**読者の反応:**
1. 「すごい！試してみよう」
2. （実際に試す）
3. 「あれ、10倍速くならない...」
4. 「著者は嘘をついていた」
5. **信頼を失う**

## 本書の特徴

### 1. 誠実性を最優先

本書は、**誠実で正確な情報**を提供することを最優先にしています。

- 実測していないデータは「実測」として書かない
- 架空の数値を事実のように書かない
- 情報源を明記する
- 事実と推測を明確に区別する

**MIT修士論文レベルの品質基準**に準拠しています。

### 2. 実践的な例

抽象的な説明ではなく、**実際に使える例**を豊富に掲載しています。

- 実際のOSSプロジェクト（claude-code-skills、zenn-articlesなど）の例
- 想定されるケースを明確に示した例
- Before/Afterの比較

### 3. 理論と実践のバランス

**なぜそうすべきか（理論）** と **どうすればよいか（実践）** の両方を説明します。

理論だけでは実務で使えません。実践だけでは応用が利きません。両方を学ぶことで、どんな状況でも適切なドキュメントを作成できるようになります。

### 4. チェックリスト付き

各章の最後には、**実践的なチェックリスト**があります。

自分のドキュメントを評価し、改善点を見つけられます。

## さあ、始めましょう

優れたドキュメントは、優れたソフトウェアの必須条件です。

ドキュメント作成は、コーディングと同じく**学習可能なスキル**です。誰でも、練習すれば上達します。

本書は、あなたのドキュメント作成スキルを、**初心者から実践者、そしてチームのリーダー**へと引き上げるための完全ガイドです。

**最も重要なのは、誠実であること。**

検証していないことを事実のように書かず、読者の信頼を第一に考える。この原則さえ守れば、あなたのドキュメントは必ず価値あるものになります。

それでは、第1章「ドキュメント作成の原則」から始めましょう。
