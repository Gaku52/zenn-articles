## 【重要】Goには「参照渡し」が存在しない

他のプログラミング言語を経験している方は、ここを必ず読んでください。

**Goの引数渡しは、常に「値渡し（pass by value）」のみです。** 参照渡しという概念はGoに存在しません。

### 他言語との比較

| 言語 | 値渡し | 参照渡し |
|------|--------|---------|
| C++ | ✅ | ✅（`int&`） |
| Java | ✅（プリミティブ） | 参照の値渡し（オブジェクト） |
| Python | — | 参照の値渡し（すべてオブジェクト） |
| Ruby | — | 参照の値渡し（すべてオブジェクト） |
| **Go** | **✅（これだけ）** | **❌ 存在しない** |

### 「ポインタ渡し」は「参照渡し」ではない

```go
func double(n *int) {
    *n = *n * 2
}

func main() {
    x := 10
    double(&x)
}
```

このコードを見て「参照渡しをしている」と感じるかもしれませんが、正確には：

- **渡しているのは `x` のメモリアドレスという「値」**
- ポインタという値をコピーして渡している = 値渡し
- 関数内でそのアドレスを経由して元の変数を書き換えているだけ

> 「ポインタ渡し」と「参照渡し」は、**結果が似て見えても仕組みが違います。**
> Goで「参照渡し」という言葉を使うと技術的に正確ではないため、「ポインタを渡す」と表現します。

---

## なぜポインタを学ぶのか

Goの関数は**常に値をコピーして渡します**（値渡し）。これは安全ですが、2つの問題があります：

1. **大きな構造体のコピーはコストが高い**
2. **関数内で元の値を変更したい場合がある**

ポインタはこの問題を解決します。「値そのもの」ではなく「値がある場所（メモリアドレス）」を渡すことで、コピーなしに元の値にアクセスできます。

---

## ポインタの基本

```go
x := 42

p := &x          // & でアドレスを取得
fmt.Println(p)   // 0xc0000b6010（メモリアドレス）
fmt.Println(*p)  // 42（* で値を取り出す = 間接参照）

*p = 100         // ポインタ経由で値を変更
fmt.Println(x)   // 100（元の変数も変わる）
```

### 図解

```
変数 x:    42
           ↑
ポインタ p: 0xc0000b6010  ──→ メモリの場所を指している

&x  →  x のアドレス（ポインタ）を取得
*p  →  ポインタが指す先の値を取得
```

### ポインタの型

```go
var x int = 42
var p *int = &x    // *int 型（int へのポインタ）

var s string = "hello"
var ps *string = &s  // *string 型（string へのポインタ）
```

`*型` がポインタ型です。`*int` は「int型の値を指すポインタ」を意味します。

## 値渡し vs ポインタ渡し

### 値渡し（デフォルト）

```go
func double(n int) {
    n = n * 2  // コピーを変更しているだけ
}

func main() {
    x := 10
    double(x)
    fmt.Println(x)  // 10（変わらない）
}
```

### ポインタ渡し

```go
func double(n *int) {
    *n = *n * 2  // ポインタ経由で元の値を変更
}

func main() {
    x := 10
    double(&x)
    fmt.Println(x)  // 20（変わった）
}
```

### 実用的な例

```go
// ユーザー情報を更新する関数
type User struct {
    Name string
    Age  int
}

// ポインタ渡しで元の構造体を変更
func birthday(u *User) {
    u.Age++
}

func main() {
    user := User{Name: "Alice", Age: 29}
    birthday(&user)
    fmt.Println(user.Age)  // 30
}
```

> **補足**: 構造体のポインタでは `(*u).Age` と書く必要がありますが、Goは `u.Age` と書いても自動で間接参照してくれます。これは便利な構文糖衣です。

## スライスとマップの落とし穴

スライスとマップは「参照のように動く」ため、最も混乱を招きやすい型です。

### スライスの挙動

```go
func modifySlice(s []int) {
    s[0] = 999           // ① 既存要素の変更 → 呼び出し元に反映される
    s = append(s, 100)   // ② append による追加 → 呼び出し元には反映されない
}

func main() {
    nums := []int{1, 2, 3}
    modifySlice(nums)
    fmt.Println(nums)  // [999 2 3]（① は反映、② は反映されない）
}
```

**なぜこうなるのか？**

スライスの実体は「ヘッダ」と呼ばれる小さな構造体です：

```
スライスヘッダ（値渡しでコピーされる）
┌─────────────────┐
│ ptr → [1, 2, 3] │  ← 配列へのポインタ（同じ配列を指している）
│ len = 3         │
│ cap = 3         │
└─────────────────┘
```

- `s[0] = 999` はポインタの先の配列を書き換えるため、元のスライスにも反映される
- `append` はヘッダ（len/cap）を書き換えるだけで、コピーされたヘッダを変えても元には影響しない

**スライス自体を変更したいときはポインタを使う：**

```go
func appendValue(s *[]int, v int) {
    *s = append(*s, v)
}

func main() {
    nums := []int{1, 2, 3}
    appendValue(&nums, 100)
    fmt.Println(nums)  // [1 2 3 100]
}
```

### マップの挙動

```go
func addEntry(m map[string]int) {
    m["new"] = 42  // マップ本体は共有されているため呼び出し元にも反映
}

func main() {
    scores := map[string]int{"alice": 90}
    addEntry(scores)
    fmt.Println(scores)  // map[alice:90 new:42]
}
```

マップはヘッダの中にポインタを持つため、要素の追加・変更は呼び出し元に反映されます。ただし、これも「マップというヘッダの値を渡している」であり、参照渡しではありません。

### まとめ

| 型 | 既存要素の変更 | 再代入・append |
|----|-------------|--------------|
| int, string, struct | 呼び出し元に反映されない | 反映されない |
| スライス | **反映される**（内部配列が共有） | 反映されない |
| マップ | **反映される**（内部データが共有） | 反映されない |
| ポインタ | **反映される** | **反映される** |

## ポインタを使う場面・使わない場面

### ポインタを使う場面

```go
// 1. 関数内で元の値を変更したい
func reset(s *[]int) {
    *s = (*s)[:0]
}

// 2. 大きな構造体のコピーを避けたい
type LargeConfig struct {
    // たくさんのフィールド...
}
func processConfig(cfg *LargeConfig) { ... }

// 3. nilを使って「値がない」ことを表現したい
func findUser(id int) *User {
    // 見つからない場合は nil を返す
    return nil
}
```

### ポインタを使わない場面

```go
// 1. 小さな値（int, bool, string など）
func add(a, b int) int {  // ポインタにする必要なし
    return a + b
}

// 2. スライスやマップ（内部的にポインタを含む）
func process(items []string) {  // スライスは内部的にポインタを持つ
    // 既存要素の変更は呼び出し元にも反映される
    // ただし append による拡張はスライスヘッダのコピーには反映されない
}

// 3. 値を変更する必要がない場合
func display(u User) {  // 読み取りだけなら値渡しで安全
    fmt.Println(u.Name)
}
```

### 判断基準

| 条件 | 値渡し | ポインタ渡し |
|------|--------|-------------|
| 小さな型（int, bool, string） | ✅ | - |
| 大きな構造体（フィールド多数） | - | ✅ |
| 関数内で値を変更したい | - | ✅ |
| スライス、マップ | ✅ すでに参照 | - |
| nil を使いたい | - | ✅ |

## nil ポインタ

ポインタのゼロ値は `nil` です。

```go
var p *int  // nil
fmt.Println(p)  // <nil>

// nil ポインタの間接参照はパニック（ランタイムエラー）
// fmt.Println(*p)  // panic: runtime error: invalid memory address

// 使う前にnilチェック
if p != nil {
    fmt.Println(*p)
}
```

---

## new 関数

`new(T)` はゼロ値で初期化された `T` へのポインタを返します。

```go
p := new(int)     // *int, 値は 0
fmt.Println(*p)   // 0

s := new(string)  // *string, 値は ""
fmt.Println(*s)   // ""
```

ただし実務では `new` よりも `&` を使うことが多いです：

```go
// new を使う場合
p := new(User)

// & を使う場合（こちらが一般的）
p := &User{Name: "Alice", Age: 30}
```

---

## ポインタの制約

Goのポインタは安全に設計されています。C言語のような危険な操作はできません：

```go
// ❌ ポインタ演算はできない
// p++
// p + 1

// ❌ 型の異なるポインタ間のキャストはできない
// (*float64)(p)

// ✅ ポインタにできるのは & と * だけ
```

これにより、メモリ安全性を保ちながらポインタの利便性を享受できます。
