## Goはなぜ生まれたのか

2007年、Google社内で3人のエンジニアが一つの課題に直面していました。

- **Robert Griesemer** — V8 JavaScript エンジンの開発者
- **Rob Pike** — Unix、UTF-8の共同設計者
- **Ken Thompson** — Unixの生みの親、C言語の共同設計者

彼らの課題は「**大規模ソフトウェア開発における複雑性の爆発**」でした。

- C++ はコンパイルが遅すぎる（Googleの大規模コードベースで数十分〜数時間）
- Java は冗長すぎる（ボイラープレートだらけ）
- Python は遅すぎる（動的型付けで型安全性もない）
- どの言語も並行処理が難しすぎる

「**シンプルで、速くて、並行処理が簡単な言語**」— これがGoの出発点です。

## Goの3つの柱

### 1. シンプルさ

Goには**意図的に省かれた機能**が多くあります：

| 他の言語にあるもの | Goの代替 |
|-------------------|---------|
| クラス継承 | 構造体の埋め込み（コンポジション） |
| 例外（try/catch） | 複数戻り値による明示的エラー処理 |
| 三項演算子（`? :`） | if文で書く |
| while文 | for文で統一 |
| 演算子オーバーロード | 関数で明示的に書く |

これは制限ではなく設計です。「機能を追加するのは簡単だが、削除するのは不可能」— だからGoは最初から少なく始めました。

### 2. 高速なコンパイル

Goのコンパイルは**秒単位**です。数万行のプロジェクトでも数秒でビルドが完了します。

```bash
$ time go build ./...
real    0m1.234s   # C++なら数分〜数時間かかるプロジェクトでも秒で終わる
```

この速さの理由：
- パッケージの依存関係が**有向非巡回グラフ（DAG）**になるよう設計されている
- 未使用のインポートは**コンパイルエラー**（無駄な依存を強制排除）
- コンパイラ自身がGoで書かれている

### 3. 組み込みの並行処理

Goでは並行処理が**言語レベル**でサポートされています。他の言語のようにライブラリやフレームワークに頼る必要がありません。

```go
package main

import (
    "fmt"
    "time"
)

func sayHello(name string) {
    time.Sleep(100 * time.Millisecond)
    fmt.Printf("Hello, %s!\n", name)
}

func main() {
    // go キーワードをつけるだけで並行実行
    go sayHello("Alice")
    go sayHello("Bob")
    go sayHello("Charlie")

    time.Sleep(200 * time.Millisecond)
}
```

`go` キーワード1つで並行実行。これがGoの並行処理です。詳しくは実践ガイドで学びますが、この簡潔さがGoの大きな魅力です。

## 他の言語との比較

| 特徴 | Go | Python | Java | Rust |
|------|-----|--------|------|------|
| 型付け | 静的 | 動的 | 静的 | 静的 |
| コンパイル | 超高速 | 不要 | 中速 | 遅い |
| 実行速度 | 高速 | 遅い | 高速 | 最速 |
| メモリ管理 | GC | GC | GC | 所有権 |
| 並行処理 | goroutine | asyncio | Thread/Virtual Thread | async/tokio |
| 学習コスト | 低い | 低い | 中程度 | 高い |
| バイナリ配布 | 単一バイナリ | ランタイム必要 | JVM必要 | 単一バイナリ |

### Goが特に強い領域

- **Webサーバー / API** — net/httpが標準搭載、高い並行性能
- **CLIツール** — 単一バイナリでクロスコンパイル可能
- **マイクロサービス** — Docker/Kubernetesとの親和性（どちらもGoで書かれている）
- **インフラツール** — Docker, Kubernetes, Terraform, Hugo... すべてGo製

### Goが苦手な領域

- **GUI アプリケーション** — 標準ライブラリにGUIサポートがない
- **数値計算・機械学習** — Python（NumPy/PyTorch）のエコシステムに及ばない
- **組み込みシステム** — Rust や C の方が適している
- **スクリプティング** — 短いスクリプトにはPythonの方が手軽

## Go Proverbs（Go格言）

Rob Pike が提唱した Go の設計哲学を表す格言です。今は全てを理解する必要はありませんが、学習を進めるにつれて「ああ、これか」と腑に落ちるようになります。

| 格言 | 意味 |
|------|------|
| **Clear is better than clever.** | 賢いコードより明快なコードを書け |
| **Errors are values.** | エラーは特別なものではなく、普通の値として扱え |
| **Don't panic.** | panicは本当に回復不能な状況だけで使え |
| **Make the zero value useful.** | 初期化しなくても使える構造体を設計せよ |
| **A little copying is better than a little dependency.** | 小さな依存を増やすくらいなら、少しのコピーの方がいい |

これらの格言は本書を通して繰り返し登場します。Goを学ぶということは、この考え方を身につけることでもあります。
