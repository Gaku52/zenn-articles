# 探索的テスト：構造化された学習とテスト

## 探索的テストとは

探索的テスト（Exploratory Testing）は、James Bachによって体系化されたテスト手法で、テスト設計、実行、学習を同時に行う構造化されたアプローチです[^1]。事前に詳細なテストケースを作成するのではなく、テスターの経験と直感を活かしながら、システムを探索的に調査します。

## 探索的テストとアドホックテストの違い

両者はしばしば混同されますが、明確な違いがあります：

| 項目 | 探索的テスト | アドホックテスト |
|------|------------|---------------|
| 構造 | 構造化されている | 非構造化 |
| 目的 | 学習と発見 | 単なるバグ発見 |
| ドキュメント | セッションノート記録 | ドキュメントなし |
| 再現性 | 高い | 低い |
| スキル要求 | 高い | 低〜中 |

探索的テストは、「考えながらテストする」という点でアドホックテストと似ていますが、体系的なアプローチと記録を伴う点で異なります。

## セッションベーステストマネジメント（SBTM）

探索的テストを管理可能にするフレームワークがSBTM（Session-Based Test Management）です[^2]。

### セッションの基本構造

**セッション時間**: 60〜120分（推奨90分）

**セッションチャーター（憲章）**
```markdown
# セッションチャーター

**目的**: 決済機能のエラーハンドリングを探索する

**探索領域**:
- クレジットカード決済
- QRコード決済

**テスト戦略**:
- 無効なカード情報
- ネットワークエラー
- タイムアウト
- 同時実行

**成果物**:
- バグレポート
- 疑問点リスト
- セッションノート
```

### セッションノートのテンプレート

```markdown
# セッションノート

**セッションID**: ET-2026-03-01-001
**テスター**: 山田太郎
**日時**: 2026/03/01 14:00-15:30
**チャーター**: 決済機能のエラーハンドリングを探索

## 時間配分
- テスト実行: 70% (63分)
- バグ報告: 20% (18分)
- セットアップ: 10% (9分)

## テストした内容
1. 有効期限切れカードでの決済 → エラー表示確認
2. CVCコード不正での決済 → エラーメッセージ適切
3. ネットワーク切断時の挙動 → リトライ機能なし（改善点）
4. 決済中のブラウザバック → 二重決済の可能性（バグ）

## 発見したバグ
- BUG-001: 決済中のブラウザバックで二重決済発生
- BUG-002: タイムアウト時のエラーメッセージが不明瞭

## 疑問・調査事項
- Q1: 決済失敗時のリトライポリシーは？
- Q2: 決済ログはどこまで保存される？

## 未探索領域
- 高額決済（100万円以上）
- 複数決済方法の切り替え

## 次回のフォーカス
- 決済ログの検証
- 高額決済のテスト
```

## 探索的テストの技法

### 1. ツアーテスト

アプリケーションを「観光ツアー」のように巡る技法です。

**ガイドツアー**: ハッピーパスを辿る
```
ユーザー登録 → ログイン → 商品検索 → カート追加 → 決済
```

**トラブルツアー**: 意図的に問題を起こす
```
無効な入力 → セッションタイムアウト → ネットワーク切断
```

**裏通りツアー**: 通常使われない機能を探索
```
管理画面 → 古いAPI → 非推奨機能
```

**美術館ツアー**: UIとUXに焦点
```
レイアウト崩れ → 色のコントラスト → アクセシビリティ
```

### 2. ペルソナベーステスト

特定のユーザーペルソナになりきってテストします。

```markdown
## ペルソナ: 高齢者ユーザー

**特性**:
- スマホ操作が不慣れ
- 視力が弱い
- ゆっくり操作

**テスト観点**:
- フォントサイズは十分か
- ボタンのタップエリアは大きいか
- エラーメッセージは分かりやすいか
- 操作のタイムアウトは適切か
```

```markdown
## ペルソナ: パワーユーザー

**特性**:
- ショートカットキーを多用
- 効率を重視
- 細かい機能まで使いこなす

**テスト観点**:
- キーボードナビゲーション
- 一括操作機能
- カスタマイズ性
```

### 3. ハーディスティック（発見的手法）

経験則に基づくテスト観点のチェックリストです。

**FCC CUTS VIDS**（James Bach考案）

- **F**eature: 機能として正しく動くか
- **C**laims: 宣伝された機能は本当か
- **C**omparable Products: 競合製品と比べてどうか
- **C**onsistency: 一貫性はあるか
- **U**sers: ユーザーは満足するか
- **T**ests: テスト自体は正しいか
- **S**tructure: 構造的に問題ないか
- **V**alue: ビジネス価値を提供するか
- **I**mage: ブランドイメージに合うか
- **D**evelopment: 開発プロセスは適切か
- **S**tandards: 標準規格に準拠しているか

### 4. CRUD マトリックス

データ操作（Create, Read, Update, Delete）の組み合わせをテストします。

| 機能 | Create | Read | Update | Delete |
|------|--------|------|--------|--------|
| ユーザー管理 | ✓ | ✓ | ✓ | ✓ |
| 商品マスタ | ✓ | ✓ | ✓ | ✓ |
| 注文履歴 | ✓ | ✓ | × | × |

各操作の境界条件や組み合わせを探索的にテストします。

## 探索的テストに適したシナリオ

### 1. 新機能の初期評価

仕様が固まっていない段階で、機能の挙動を理解するために使用します。

### 2. リグレッションテストの補完

自動化されたリグレッションテストでカバーしきれない部分を探索します。

### 3. ユーザビリティの評価

実際のユーザー視点で、使いやすさを評価します。

### 4. セキュリティテスト

攻撃者の視点で、脆弱性を探索します。

### 5. 統合テスト

複数のコンポーネントの連携を探索的に確認します。

## 探索的テストのベストプラクティス

### 1. 明確なチャーターを設定

漠然と「探索する」のではなく、焦点を絞ったチャーターを設定します。

```markdown
# 良いチャーター
「決済エラー時のユーザー体験を評価する」

# 悪いチャーター
「アプリを使ってみる」
```

### 2. 時間を区切る（タイムボックス）

セッションは90分程度に区切り、集中力を維持します。

### 3. ペアテスティング

2人1組でテストすることで、多角的な視点と即座のディスカッションが期待されます。

```
ドライバー: 実際に操作する人
ナビゲーター: 観察して指示する人

30分ごとに役割交代
```

### 4. リアルタイムノート

後からまとめるのではなく、テスト中にリアルタイムで記録します。

### 5. バグの再現手順を記録

探索中に見つけたバグは、必ず再現手順を記録します。

```markdown
## BUG-123: 決済画面でブラウザバックすると二重決済

**再現手順**:
1. 商品をカートに追加
2. 決済画面に進む
3. クレジットカード情報を入力
4. "購入確定"をクリック
5. 処理中にブラウザの戻るボタンをクリック
6. もう一度"購入確定"をクリック

**期待結果**:
- 二重決済されない
- または、既に処理済みのエラーが表示される

**実結果**:
- 2回決済が実行される

**環境**:
- Chrome 120
- Windows 11
```

## 探索的テストのツール

### マインドマップツール

探索の計画と記録にマインドマップを活用します。

- XMind
- MindMeister
- Coggle

### セッション記録ツール

- Rapid Reporter: セッションノート記録専用ツール
- TestBuddy: セッションベーステストマネジメント用

### スクリーンレコーディング

テストセッションを録画しておくと、後から振り返りができます。

- OBS Studio

- QuickTime（Mac）

## 探索的テストのメトリクス

探索的テストの効果を測定するメトリクスです。

```
カバレッジ率 = 探索したエリア / 全体エリア × 100

バグ検出率 = 探索的テストで発見したバグ数 / 総バグ数 × 100

セッション効率 = バグ発見数 / セッション数

テスト vs バグ時間比 = テスト時間 / バグ報告時間
```

理想的なテスト vs バグ時間比は 75:25 程度と言われています。

## 探索的テストとスクリプトテストのバランス

両者は対立するものではなく、補完し合う関係です。

### 推奨されるバランス

| プロジェクトフェーズ | スクリプトテスト | 探索的テスト |
|------------------|----------------|------------|
| 初期開発 | 30% | 70% |
| 安定期 | 60% | 40% |
| リグレッション | 80% | 20% |

### 組み合わせ戦略

1. **自動化テストで基本的な動作を保証**
2. **探索的テストで新しいバグや使いやすさの問題を発見**
3. **発見したバグのテストケースを自動化に追加**

このサイクルを回すことで、テストの網羅性と効率性の両立が期待されます。

## まとめ

探索的テストは、テスターのスキルと経験を最大限活用するための構造化されたアプローチです。セッションベーステストマネジメントを使って管理可能にし、ツアーテストやペルソナベーステストなどの技法を組み合わせることで、効果的なテストが期待されます。

スクリプトテストだけでは発見できない問題を見つけ出すために、探索的テストをテスト戦略に組み込むことが推奨されます。

次章では、リグレッションテストの効率的な運用方法について解説します。
