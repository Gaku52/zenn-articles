# QAプロセス全体像：開発ライフサイクルとQA活動

## QAプロセスの位置づけ

QAプロセスは、ソフトウェア開発ライフサイクル（SDLC: Software Development Life Cycle）全体に統合されます。従来のウォーターフォール開発では、テストフェーズは実装完了後に行われていましたが、現代のアジャイル開発やDevOpsでは、QA活動は開発の最初から最後まで継続的に実施されます。

これは「Shift Left Testing」と呼ばれるアプローチで、品質問題をより早い段階で発見・修正することで、全体的なコストとリスクを削減します[^1]。

## QAプロセスの6つのフェーズ

QA活動は大きく6つのフェーズに分けられます。それぞれのフェーズで実施すべき活動と成果物を見ていきましょう。

### Phase 1: 要求分析とテスト計画

#### 主な活動
- 要求仕様書、ユーザーストーリーのレビュー
- テスト可能性の評価
- 受け入れ基準（Acceptance Criteria）の明確化
- テスト戦略の策定
- テスト計画書の作成

#### 成果物
- テスト計画書
- テスト戦略ドキュメント
- リスク分析レポート
- テストスケジュール

#### チェックポイント
- すべての要求事項がテスト可能か
- 非機能要件（性能、セキュリティなど）も明確か
- リスクの高い領域は特定されているか
- テストリソース（人員、環境、ツール）は確保できるか

### Phase 2: テスト設計

#### 主な活動
- テストケースの設計
- テストデータの準備
- テスト環境の設計
- 自動化対象の選定

#### 成果物
- テストケース仕様書
- テストデータセット
- 環境構成図
- 自動化スクリプト

#### テスト設計技法
- 同値分割法：入力値を有効クラスと無効クラスに分類
- 境界値分析：境界付近の値をテスト
- デシジョンテーブル：条件の組み合わせを網羅
- 状態遷移テスト：システムの状態変化を検証
- ペアワイズ法：組み合わせを効率的に削減

### Phase 3: テスト環境構築

#### 主な活動
- テスト環境のセットアップ
- テストデータの投入
- テストツールの設定
- 環境の動作確認

#### 成果物
- 環境構築手順書
- 環境設定ファイル
- 疎通確認結果

#### 推奨される環境構成
- **開発環境（Dev）**：開発者個人の作業環境
- **統合環境（Integration）**：機能統合テスト用
- **QA環境（QA/Staging）**：QAチームによる総合テスト用
- **本番環境（Production）**：実際のサービス提供環境

各環境は可能な限り本番に近い構成とすることが推奨されます。Infrastructure as Code（IaC）ツールを活用することで、環境の一貫性と再現性を確保できます。

### Phase 4: テスト実行

#### 主な活動
- テストケースの実行
- バグの報告と追跡
- リグレッションテスト
- 探索的テスト
- パフォーマンステスト
- セキュリティテスト

#### 成果物
- テスト実行結果レポート
- バグレポート
- カバレッジレポート

#### テスト実行のベストプラクティス
- テストは独立して実行可能にする
- 失敗したテストは再現可能な状態で報告する
- テスト結果は自動的に記録・集計する
- Critical pathから優先的にテストする
- 毎日のスモークテストでビルドの健全性を確認する

### Phase 5: バグトリアージと修正確認

#### 主な活動
- バグの優先順位付け
- 開発チームとの調整
- 修正内容のレビュー
- 修正確認テスト（Verification）
- リグレッション影響の確認

#### バグ深刻度の分類例

| 深刻度 | 定義 | 対応目安 |
|--------|------|---------|
| Critical | システムダウン、データ損失 | 即時対応 |
| High | 主要機能が使用不可 | 24時間以内 |
| Medium | 一部機能に制限あり | 3日以内 |
| Low | 軽微な問題、回避策あり | 次回リリース |

#### バグトリアージのポイント
- ビジネスインパクトを考慮する
- ユーザー影響範囲を評価する
- 修正コストと残存リスクを比較する
- セキュリティ関連は優先度を上げる

### Phase 6: リリース判定とレトロスペクティブ

#### 主な活動
- Exit Criteriaの確認
- リリース判定会議（Go/No-Go Meeting）
- リリースノートのレビュー
- 本番デプロイメントの監視
- レトロスペクティブの実施

#### Exit Criteriaの例
- すべてのCritical/Highバグが解決済み
- テストカバレッジが80%以上
- パフォーマンス要件を満たしている
- セキュリティスキャンでCritical脆弱性なし
- 受け入れテスト完了

#### レトロスペクティブの観点
- 今回のリリースで上手くいったこと
- 改善が必要なこと
- 次回試してみたいこと
- 発見したバグの傾向分析
- テストプロセスの効率化案

## アジャイル開発におけるQAプロセス

アジャイル開発では、2週間程度のスプリント内でQAサイクル全体を回します。

### スプリント内のQA活動

**スプリント計画**
- ユーザーストーリーの受け入れ基準確認
- テスト可能性の評価
- テストタスクの見積もり

**スプリント中**
- 継続的なテスト実行
- 開発と並行したテストケース作成
- 毎日のスタンドアップでテスト進捗共有
- ペアテスティング、モブテスティング

**スプリントレビュー**
- 完成した機能のデモ
- 受け入れテスト
- ステークホルダーフィードバック

**スプリントレトロスペクティブ**
- QAプロセスの振り返り
- 改善アクションの決定

### 継続的インテグレーション/継続的デリバリー（CI/CD）との統合

CI/CDパイプラインにQAプロセスを組み込むことで、自動化と迅速なフィードバックを実現します：

```yaml
# CI/CDパイプライン例
stages:
  - build
  - unit-test
  - integration-test
  - deploy-staging
  - e2e-test
  - security-scan
  - performance-test
  - deploy-production
  - smoke-test
```

各ステージで品質ゲートを設け、基準を満たさない場合は次のステージに進まないようにします。

## QAプロセスの継続的改善

QAプロセスは一度確立して終わりではなく、継続的に改善していく必要があります。

### 改善サイクル（PDCA）

**Plan（計画）**
- 品質メトリクスから課題を特定
- 改善目標を設定
- 改善策を立案

**Do（実行）**
- 小規模にパイロット実施
- チーム内で試行

**Check（評価）**
- メトリクスで効果測定
- チームからフィードバック収集

**Act（改善）**
- 成功した施策を標準化
- 失敗から学び次の改善へ

### 品質成熟度の向上

組織の品質成熟度は段階的に向上します：

**レベル1：初期段階**
- アドホックなテスト
- 属人的なプロセス

**レベル2：管理段階**
- テスト計画書を作成
- バグトラッキング開始

**レベル3：定義段階**
- 標準化されたプロセス
- メトリクス収集開始

**レベル4：定量的管理段階**
- データドリブンな意思決定
- プロセスの定量的管理

**レベル5：最適化段階**
- 継続的改善が文化に
- 自動化と効率化の追求

## まとめ

QAプロセスは開発ライフサイクル全体に統合され、6つのフェーズ（要求分析、テスト設計、環境構築、テスト実行、バグ管理、リリース判定）を通じて品質を保証します。アジャイル開発やCI/CDとの統合により、より迅速かつ継続的な品質保証が期待されます。

次章では、品質を定量的に測定するための品質メトリクスの基礎について解説します。
