# テスト計画書テンプレート：実務で使える計画立案

## テスト計画書の目的

テスト計画書は、テスト活動の全体像を文書化し、チーム全体で合意を得るための重要なドキュメントです。計画書を作成することで、以下の効果が期待されます：

- テスト範囲とアプローチの明確化
- リソースとスケジュールの可視化
- リスクと対策の共有
- ステークホルダーとの認識合わせ
- プロジェクト終了後の振り返り資料

## テスト計画書の構成

IEEE 829-2008[^1]をベースに、実務で使いやすい構成を紹介します。

### 1. 概要（Overview）

#### 1.1 プロジェクト概要

```markdown
## プロジェクト概要

**プロジェクト名**: ECサイトリニューアル v2.0
**目的**: ユーザー体験の向上と決済機能の強化
**対象リリース**: 2026年3月15日
**文書作成者**: QAチーム 山田太郎
**最終更新日**: 2026年2月1日
```

#### 1.2 テスト対象

```markdown
## テスト対象

### スコープ内
- 新規決済機能（クレジットカード、QR決済）
- 商品検索機能の改善
- レスポンシブデザイン対応
- 管理画面の新機能

### スコープ外
- 既存の会員管理機能（変更なし）
- レガシーAPI（廃止予定）
- 過去の注文履歴（移行のみ）
```

### 2. テスト戦略（Test Strategy）

#### 2.1 テストレベル

各テストレベルの配分と責任者を明記します。

```markdown
## テストレベル

| テストレベル | 割合 | 実施者 | ツール |
|------------|------|--------|--------|
| ユニットテスト | 60% | 開発者 | Jest, pytest |
| 統合テスト | 25% | 開発者・QA | Supertest |
| E2Eテスト | 10% | QA | Playwright |
| 手動探索テスト | 5% | QA | - |
```

#### 2.2 テストタイプ

```markdown
## テストタイプ

### 機能テスト
- 新規機能の動作確認
- 既存機能のリグレッション
- エラーハンドリング

### 非機能テスト
- **性能テスト**: 1000同時接続でレスポンスタイム2秒以内
- **セキュリティテスト**: OWASP Top 10チェック
- **アクセシビリティテスト**: WCAG 2.1 Level AA準拠
- **互換性テスト**: Chrome, Safari, Firefox最新2バージョン

### その他
- ユーザビリティテスト: 5名のベータユーザーでの検証
- ローカライゼーションテスト: 日本語・英語
```

#### 2.3 リスク分析

リスクベースでテストの優先順位を決定します。

```markdown
## リスク分析

| 機能 | 影響度 | 発生確率 | リスクレベル | テスト優先度 |
|------|--------|---------|------------|------------|
| 決済処理 | Critical | Medium | High | P0 - 最優先 |
| 商品検索 | High | Low | Medium | P1 - 高 |
| UI改善 | Medium | Low | Low | P2 - 中 |
```

**リスク軽減策**:
- 決済処理: 専任テスターを配置、テストカード環境で徹底検証
- 本番環境: カナリアリリースで段階的展開
- ロールバック計画の策定

### 3. テスト環境（Test Environment）

#### 3.1 環境構成

```markdown
## テスト環境

### 開発環境（Dev）
- URL: https://dev.example.com
- データベース: PostgreSQL 15（開発用データ）
- 用途: 開発者の機能確認

### QA環境（Staging）
- URL: https://staging.example.com
- データベース: PostgreSQL 15（本番相当データ）
- 用途: QAチームによる総合テスト
- 本番との差異: 決済は sandbox環境

### 本番環境（Production）
- URL: https://www.example.com
- データベース: PostgreSQL 15（冗長構成）
```

#### 3.2 テストデータ

```markdown
## テストデータ

### 準備方法
- マスターデータ: 本番DBの匿名化スナップショット
- トランザクションデータ: 自動生成スクリプト
- 個人情報: フェイカーライブラリで生成

### データセット
- 正常系ユーザー: 100件
- 異常系データ: 境界値、不正データ各50件
- 大量データ: 性能テスト用100万レコード
```

### 4. スケジュール（Schedule）

#### 4.1 マイルストーン

```markdown
## テストスケジュール

| フェーズ | 開始日 | 終了日 | 成果物 |
|---------|--------|--------|--------|
| テスト計画 | 2/1 | 2/5 | テスト計画書 |
| テストケース作成 | 2/6 | 2/15 | テストケース100件 |
| テスト環境構築 | 2/10 | 2/20 | 環境構築完了 |
| テスト実行（Iteration 1） | 2/21 | 2/28 | 実行結果レポート |
| バグ修正・再テスト | 3/1 | 3/10 | バグ修正完了 |
| リグレッションテスト | 3/11 | 3/13 | リグレッション完了 |
| リリース判定 | 3/14 | 3/14 | Go/No-Go判定 |
| 本番リリース | 3/15 | 3/15 | リリース完了 |
```

#### 4.2 リソース配分

```markdown
## リソース計画

### QAチーム
- QAリード: 1名（全期間）
- テストエンジニア: 3名（2/21～3/13）
- 自動化エンジニア: 1名（2/6～2/28）

### 工数見積もり
- テストケース作成: 40時間
- 自動化スクリプト作成: 60時間
- テスト実行: 120時間
- バグ検証: 40時間
- **合計**: 260時間
```

### 5. Entry/Exit Criteria

#### 5.1 Entry Criteria（テスト開始条件）

```markdown
## Entry Criteria

### 必須条件
- [ ] すべての機能実装が完了している
- [ ] ユニットテストカバレッジが70%以上
- [ ] コードレビューが完了している
- [ ] テスト環境が構築済み
- [ ] テストデータが準備済み
- [ ] Critical/Highバグが0件

### 推奨条件
- [ ] API仕様書が最新化されている
- [ ] 設計ドキュメントがレビュー済み
```

#### 5.2 Exit Criteria（テスト終了条件）

```markdown
## Exit Criteria

### 必須条件
- [ ] 計画されたテストケースの95%以上実行完了
- [ ] Critical/Highバグが0件
- [ ] Mediumバグの80%以上が解決済み
- [ ] リグレッションテストが完了
- [ ] 性能要件を満たしている
- [ ] セキュリティスキャンでCritical脆弱性なし

### リリース判定基準
- [ ] Exit Criteriaすべてを満たす
- [ ] ステークホルダーの承認取得
- [ ] ロールバック手順が確認済み
```

## テスト計画書作成のベストプラクティス

### 1. 関係者とレビューする

計画書はドラフト段階で開発チーム、プロダクトマネージャー、ステークホルダーとレビューします。認識のずれを早期に発見できます。

### 2. 現実的な見積もり

過去のプロジェクトデータを参考に、バッファを持った現実的なスケジュールを立てます。一般的に、見積もりの1.3〜1.5倍の時間がかかると言われています。

### 3. 定期的な更新

テスト計画は最初に作って終わりではなく、プロジェクトの進行に応じて定期的に更新します。変更履歴を残すことも重要です。

### 4. メトリクスの定義

計画段階で、どのようなメトリクスを収集するかを決めておきます。後から追加すると過去のデータが取得できません。

### 5. リスクベースアプローチ

すべてを均等にテストするのではなく、リスクの高い領域に重点を置きます。リソースは有限であることを認識します。

## 小規模プロジェクト向け簡易版

スタートアップや小規模チームでは、軽量なテスト計画で十分な場合もあります。

```markdown
# テスト計画（簡易版）

## スコープ
- 新機能A, B, C
- リグレッション対象: 決済周り

## スケジュール
- テスト期間: 3/1 - 3/7
- リリース: 3/10

## リソース
- QA: 田中（全テスト）
- 開発者: 佐藤（ユニットテスト）

## Exit Criteria
- すべての機能が動作する
- Criticalバグなし
- 決済の手動確認完了

## リスク
- 外部API障害時の対応確認必要
```

## まとめ

テスト計画書は、テスト活動の羅針盤です。スコープ、戦略、スケジュール、リソース、Entry/Exit Criteriaを明確にすることで、効率的かつ効果的なテストが期待されます。

プロジェクトの規模や性質に応じて、詳細度を調整することが重要です。大切なのは、チーム全体が同じ方向を向いてテスト活動を進められることです。
